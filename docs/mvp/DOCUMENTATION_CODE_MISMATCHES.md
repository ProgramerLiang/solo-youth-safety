# 文档与代码不一致点（文档可信度 / 版本漂移）

更新时间：2026-03-26

本文只记录 **README / docs 与实际代码之间已经确认的不一致点**，用于差异审查时避免“只看文档不看代码”。

## 判定口径
- **代码优先级高于说明文档**：当 README / API 文档与 `backend/main.py` 冲突时，以代码行为为准。
- **任务清单优先级次高**：`docs/mvp/TASKS.md` 可用于判断“团队是否已经意识到该漂移问题，但尚未修正”。
- **本表不评价需求取舍**：这里只记录“写了什么”与“实际实现了什么/暴露了什么”的差异。

## 1. 版本口径需要显式拆分：前端产物是 0.3.0，后端当前 OpenAPI 仍是 0.2.1

### 当前文档表述
- `README.md`、`backend/README.md`、`frontend/README.md` 的正确治理方向，不是把所有端统一写成同一个版本号，而是**如实区分版本语义**：
  - 前端 / Android 产物版本基线：`0.3.0`
  - 后端当前 OpenAPI / FastAPI 暴露版本：`0.2.1`
- 文档若提到统一版本策略，应明确那是**后续收口目标**。
- 文档若提到 `SAFETY_APP_VERSION`，也必须明确它目前只是规划中的配置入口，**不是现有代码能力**。

### 实际代码与产物边界
- `backend/main.py` 当前 FastAPI `version` 仍来自硬编码常量 `APP_VERSION = "0.2.1"`。
- 后端代码当前只读取 `SAFETY_DB_PATH`，**并没有读取 `SAFETY_APP_VERSION`**。
- 前端 / Android 产物版本口径则仍以前端版本基线 `0.3.0` 为主。

### 结论
当前第一个真实漂移，**不再是“README / backend README 仍然写错成 0.3.0”**，而是：
- 前端与 Android 产物版本基线：`0.3.0`
- 后端 OpenAPI 默认版本：`0.2.1`
- “后端版本可通过 `SAFETY_APP_VERSION` 覆盖”的说法若继续出现在文档中，将重新构成失真

也就是说，文档现在需要重点说明的是 **跨端版本尚未统一，且后端版本号尚无环境变量覆盖能力**。

### 影响
- Swagger / OpenAPI 中看到的后端版本号，会与 APK/AAB 文件名、Android `versionName`、前端注入版本不同。
- 审查时应把这视为“前后端版本同步策略尚未收口”。
- 若有人把 `SAFETY_APP_VERSION` 当成现状能力，会高估当前后端的版本配置治理完成度。

## 2. 安全与 CORS 已完成 MVP 级最小基线，但仍不等于正式账号体系

### 当前文档表述
- `docs/api/README.md`、`README.md`、`backend/README.md` 现已统一表述：远端身份请求头、头/体一致性校验、跨用户资源拒绝与 CORS 白名单，已作为 **MVP 级最小授权边界** 落地。
- 同时这些文档也明确保留边界声明：当前**仍未**实现正式登录、token、session、API key、OAuth/JWT、多租户组织模型或共享授权流。

### 实际代码
`backend/main.py` 当前已不再是完全开放开发态，而是具备以下最小收口：
- CORS 来源改为从 `SAFETY_ALLOWED_ORIGINS` 读取白名单，而非默认 `allow_origins=["*"]`
- 受保护接口要求 `X-Safety-User-Id`，部分写接口还要求 `X-Safety-Device-Id`
- 请求头身份若与 query / body 中的 `userId`、`deviceId` 不一致，会返回 4xx 拒绝
- 联系人等资源已增加最小归属校验，跨用户读取/删除会被拒绝
- 仍未实现正式登录、token、session、API key 等完整鉴权体系

### 结论
当前第二个真实差异，已经不再是“文档写了但代码完全没做”，而是：
- **MVP 级最小安全边界已在代码中生效**
- 但它仍然只是远端联调 / MVP 试用基线，**不能等同于正式账号体系、正式安全合规或生产级授权模型**

因此，这一项现在应被解读为：**最小安全基线已落地，但完整认证与授权体系仍是后续补洞项。**

### 影响
- 对外沟通时，可以如实表述“已具备 MVP 级最小授权边界与 CORS 白名单收口”。
- 但不能把当前能力夸大为“已完成正式鉴权”“已具备完整账号系统”或“已完成生产级安全治理”。
- 整改清单中的后端分组，仍应把 **正式认证、设备管理、审计、部署安全、共享授权流** 作为后续安全方向；不要因为最小基线落地，就误判为安全工作已经完成。

## 3. API 文档的能力范围基本可信，但只能代表 MVP 基线，不代表原始愿景能力已存在

### 文档表述
`docs/api/README.md` 当前只描述了：
- `health`
- `emergency/config`
- `sos/events`
- `tracking/points`
- `tracking/timeline`
- `contacts`

### 实际代码
`backend/main.py` 中路由与数据库表也确实只覆盖：
- 紧急配置
- 联系人
- SOS 事件
- SOS 通知日志
- 轨迹点

### 结论
这里 **文档与代码基本一致**，但要注意一个常见误读风险：
- `docs/api/README.md` 可信地反映了当前 MVP API 边界
- 不能据此反推“原始方案里提到的围栏、规则引擎、远程控制、录音拍照、偷拍检测等能力只是没写进文档”
- 更合理的判断是：**这些能力当前代码中确实不存在**

### 影响
在最终报告里，API 文档应被用作“当前能力边界”的佐证，而不是“高级能力已实现但未展示”的证据。

## 4. 任务清单已经承认“统一版本与配置入口”未完成，说明版本漂移问题并非偶发

### 文档表述
`docs/mvp/TASKS.md` 第 4 项“统一版本与配置入口”仍未完成，包括：
- 前端 API 地址改为环境变量配置
- 明确前后端版本同步策略
- 收敛 README / 前端 README / 后端 README 的环境配置说明

### 与代码/文档现状的对应关系
这与当前已发现的问题完全对上：
- 前端 / Android 产物版本与后端 OpenAPI 默认版本仍未统一
- 后端版本入口与前端版本入口并未收敛为同一套发布口径
- 多份文档虽然已较真实地描述现状，但整体版本同步策略仍未完成

### 结论
`docs/mvp/TASKS.md` 反而提升了审查可信度：
- 它证明团队**已经意识到版本与配置口径尚未完全统一**
- 但截至当前代码，**问题仍未被真正收口**

## 审查使用建议
在最终差异审查报告中，可直接使用以下判断：

1. **当前主要版本差异不再是 README 写错，而是前端/Android 产物版本与后端 OpenAPI 默认版本暂未同步。**
2. **安全相关表述应认定为“已落地的 MVP 最小边界 + 后续仍待补齐的正式认证体系”，不能误写成“完全未实现”，也不能夸大成“生产级安全能力”。**
3. **当前 API 文档对 MVP 能力边界总体可信，但只覆盖 MVP 基线，不应外推为原始愿景能力已实现。**
4. **任务清单已承认版本与配置入口仍在收口中，因此现阶段应避免把多端版本号视为已经统一。**

# 项目现状评估与下一阶段 Roadmap

更新时间：2026-03-22

## 一句话结论
当前项目已经完成了 **可运行、可安装、可演示的 Android MVP**，但距离 **真实用户可持续使用** 和 **对外试用交付** 还差一轮以可靠性、数据持久化、正式用户体验为核心的补强。

---

## 1. 当前已完成能力

### 产品闭环
- 首次启动自动申请定位权限
- 首次引导完成紧急配置
- 支持一键 SOS，含 5 秒倒计时与取消
- 电话号码 / 短信号码支持自定义，也支持留空
- 短信模板支持自定义与变量预览
- 已支持抽屉侧边栏多页面布局（总览 / 主题 / 配置 / 联系人 / SOS / 历史 / 工具）
- 已支持 Material 动态主题（壁纸吸色 / 预设 / 自定义调色板）
- 已支持用户 / 设备标识持久化（移除 `u_123` / `android-device-001` 硬编码）
- 已支持正式联系人管理（新增 / 编辑 / 删除 / 一键填入号码）
- 已支持 SOS 历史记录与通知详情查看
- 已支持手动刷新当前位置、2~3 次多次采样与定位精度分级提示
- 已支持前台周期轨迹写入、自动采样周期配置、弱网待补发队列
- 已支持左上角菜单按钮、页面空白处右滑呼出、左滑收起的抽屉侧边栏
- Android 内可直接发送短信并直接发起拨号（需授予权限）

### 技术实现
- `frontend`：React + Vite
- `backend`：FastAPI + SQLite
- `android`：Capacitor 6
- 已支持 **APK 内本地后端模式**，无需依赖手机内运行 Python/FastAPI 进程
- Android 本地关键数据已迁移到 **Capacitor Preferences** 持久化
- 远端 FastAPI 已完成 SQLite 持久化落盘
- Web 调试模式仍可连接远端 FastAPI

### 本地后端能力
- 已覆盖 MVP API：
  - `GET /health`
  - `GET/POST /emergency/config`
  - `POST /sos/events`
  - `POST /tracking/points`
  - `GET /tracking/timeline`
  - `GET/POST /contacts`
- 已支持本地数据面板
- 已支持联系人 / 轨迹预览卡片
- 已支持本地快照导出 / 导入
- 已支持开发者模式隐藏入口（五连击版本号后显示工具页）

### Android 交付
- 已成功构建 debug APK、signed release APK、release AAB
- debug APK 路径：`frontend/android/app/build/outputs/apk/debug/solo-youth-safety-v<version>-debug.apk`
- release APK 路径：`frontend/android/app/build/outputs/apk/release/solo-youth-safety-v<version>-release.apk`
- release AAB 路径：`frontend/android/app/build/outputs/bundle/release/solo-youth-safety-v<version>-release.aab`
- release 签名资产主目录与备份目录均已放在项目外

## 2. 当前主要短板

## 2.1 用户体验仍偏“演示版”
当前已经有 API 和数据验证能力，且主要功能已经拆成独立页面，但离正式产品体验还有差距：
- 虽已支持基础联系人管理，但仍缺少联系人角色、优先级等正式策略
- 已支持基础 SOS 历史查看，但仍缺少地图化回放与筛选能力
- 轨迹仅能列表预览，缺少地图化回放
- 调试工具虽已从主流程中分离，但当前阶段不继续优先处理其进一步隔离

## 2.2 定位与守护能力仍需继续增强
- 已支持前台周期轨迹写入策略（可配置 30 / 60 / 180 / 300 秒）
- 已支持弱网/断网失败后的本地待补发队列，并在网络恢复 / 回到前台后自动重试
- 已支持手动刷新 / SOS 前刷新时进行 2~3 次采样，并自动采用精度最佳结果
- 已支持定位精度分级提示（很准 / 良好 / 一般 / 较弱）
- 暂无 Android 前台服务或后台守护方案
- 当前周期守护仍主要覆盖 App 存活期间，后台长期追踪能力仍待评估

## 2.3 身份模型已完成本地持久化，但远端身份体系仍待补强
当前前端 / APK 侧已移除硬编码身份，首次启动会自动生成并持久化：
- `userId`
- `deviceId`

剩余问题主要在于：
- 尚未接入正式账号体系
- 远端后端仍未做鉴权
- 多设备 / 多账号切换策略未明确
- 远端数据持久化后的归属校验仍需补强

## 2.4 数据持久化可靠性仍有短板
- Android 本地关键数据已从 `localStorage` 升级到 `Capacitor Preferences`
- 远端 FastAPI 已从内存结构升级到 SQLite 落盘
- 但当前本地仍是 JSON Key-Value 方案，尚未升级为结构化数据库（如 SQLite）
- 当前远端数据库仍缺少迁移、备份、索引优化与正式鉴权配套
- App 清数据 / 浏览器清缓存后，本地数据仍可能丢失

## 2.5 安全与隐私能力不足
- 后端未做鉴权
- CORS 当前是全开放
- 本地敏感数据为明文存储
- 快照导入导出含联系人、轨迹、SOS 数据
- 调试能力和正式能力尚未隔离

## 2.6 发布能力仍需继续收口
- 已完成 release 签名配置
- 已生成 signed release APK / AAB
- 签名资产已放在项目外，并额外备份一份到项目外目录
- 仍缺少应用图标、启动图、正式发布清单与渠道分发策略

## 2.7 测试与文档同步不足
- 缺少前端单元测试
- 缺少后端接口测试
- 缺少 CI 构建检查
- 根 README / TASKS 需要持续同步实际状态

## 3. 下一阶段优先级

## P0：下一轮必须补齐
这些项决定项目能否从“演示版”进入“可试用版”。

### 3.1 联系人能力补强
- 联系人角色 / 分组 / 优先级
- 区分拨号联系人与短信联系人
- 支持从联系人直接建立通知策略

### 3.2 SOS 历史能力补强
- 时间筛选 / 数量筛选
- 地图化回放
- 通知结果筛选
- 更清晰的事件详情组织

### 3.3 远端身份与鉴权补强
- 建立账号 / 登录占位模型
- 后端增加鉴权与归属校验
- 明确多设备 / 多账号切换策略

### 3.4 存储升级
- 本地：从 `Capacitor Preferences` 继续升级为更适合复杂数据的结构化存储（如 SQLite）
- 远端：在 SQLite 基础上补齐迁移、索引、备份与后续 PostgreSQL 演进路径

### 3.5 Android 发布收口
- 图标 / 启动图 / 包名 / 版本号整理
- 渠道分发与安装说明
- 发布前核对清单
- 后续视需要再细化 debug 与 release 行为差异

### 3.6 SOS 可靠性补强
- [x] 周期轨迹写入策略（前台自动采样）
- [x] 弱网 / 离线补偿（本地待补发队列）
- [x] 多次采样与定位精度分级提示
- [ ] 更明确的定位失败恢复策略
- [ ] 后续前台服务 / 守护能力评估

### 3.7 自动化测试基础设施
- 前端核心逻辑测试
- FastAPI 接口测试
- 基础 CI（至少检查构建是否通过）

## 6. 建议按 3 个阶段推进

## 阶段 A：从“演示版”到“试用版”
目标：让非开发人员也能稳定试用。

建议内容：
1. 联系人角色与通知策略补强
2. SOS 历史筛选与地图化增强
3. 发布素材与分发说明收口
4. 远端身份 / 鉴权补强
5. 前台服务 / 守护能力评估

### 阶段 A 验收标准
- 不依赖 mock 按钮也能完整操作主要流程
- 普通用户可以管理联系人并完成 SOS 触发
- 可以查看历史 SOS 事件
- 能生成 signed release 包并稳定安装

## 阶段 B：从“试用版”到“可靠版”
目标：提高数据可靠性与真实场景可用性。

建议内容：
1. 本地结构化存储升级（Preferences → SQLite）
2. 远端身份 / 鉴权补强
3. 测试与 CI
4. 安全与隐私补强

### 阶段 B 验收标准
- App 重启 / 后端重启后关键数据不丢失
- 本地与远端模式都有基础自动化验证
- 真实移动测试中能稳定记录轨迹与 SOS 数据

## 阶段 C：从“可靠版”到“产品增强版”
目标：体现差异化价值。

建议内容：
1. 风险区域提示
2. 安全路线推荐
3. 报警事件地图回放
4. 更深层的守护体验功能

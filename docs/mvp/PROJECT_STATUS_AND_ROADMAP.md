# 项目现状评估与下一阶段 Roadmap

更新时间：2026-03-22

## 一句话结论
当前项目已经完成了 **可运行、可安装、可演示的 Android MVP**，但距离 **真实用户可持续使用** 和 **对外试用交付** 还差一轮以可靠性、数据持久化、正式用户体验为核心的补强。

---

## 1. 当前已完成能力

### 产品闭环
- 首次启动自动申请定位权限
- 首次引导完成紧急配置
- 支持一键 SOS，含 5 秒倒计时与取消
- 电话号码 / 短信号码支持自定义，也支持留空
- 短信模板支持自定义与变量预览
- 已支持正式联系人管理（新增 / 编辑 / 删除 / 一键填入号码）
- 已支持 SOS 历史记录与通知详情查看
- Android 内可拉起系统拨号与短信

### 技术实现
- `frontend`：React + Vite
- `backend`：FastAPI
- `android`：Capacitor 6
- 已支持 **APK 内本地后端模式**，无需依赖手机内运行 Python/FastAPI 进程
- Web 调试模式仍可连接远端 FastAPI

### 本地后端能力
- 已覆盖 MVP API：
  - `GET /health`
  - `GET/POST /emergency/config`
  - `POST /sos/events`
  - `POST /tracking/points`
  - `GET /tracking/timeline`
  - `GET/POST /contacts`
- 已支持本地数据面板
- 已支持联系人 / 轨迹预览卡片
- 已支持本地快照导出 / 导入

### Android 交付
- 已多次成功构建 debug APK
- APK 路径：`frontend/android/app/build/outputs/apk/debug/app-debug.apk`

## 2. 当前主要短板

## 2.1 用户体验仍偏“演示版”
当前已经有 API 和数据验证能力，但面向普通用户的正式页面还不完整：
- 虽已支持基础联系人管理，但仍缺少联系人角色、优先级等正式策略
- 已支持基础 SOS 历史查看，但仍缺少地图化回放与筛选能力
- 轨迹仅能列表预览，缺少地图化回放

## 2.2 定位与守护能力还不够强
- 暂无持续定位 / 周期定位策略
- 暂无前台服务或后台守护方案
- 暂无位置新鲜度判断
- 暂无弱网/断网补偿策略
- 暂无“手动刷新当前位置”之类的明确入口

## 2.3 身份模型仍是 demo 级
当前仍存在硬编码：
- `DEFAULT_USER = 'u_123'`
- `deviceId = 'android-device-001'`

这会影响：
- 多用户隔离
- 多设备区分
- 后续账号体系接入
- 远端数据持久化后的正确归属

## 2.4 数据持久化可靠性不足
- Android 本地后端当前依赖 `localStorage`
- FastAPI 后端当前依赖内存数据结构
- App 清数据 / 浏览器清缓存 / 服务重启后，都会丢失核心数据

## 2.5 安全与隐私能力不足
- 后端未做鉴权
- CORS 当前是全开放
- 本地敏感数据为明文存储
- 快照导入导出含联系人、轨迹、SOS 数据
- 调试能力和正式能力尚未隔离

## 2.6 发布能力仍停留在 debug
- 仅完成 debug APK
- 尚未完成 release 签名配置
- 尚未生成正式发布包（release APK / AAB）
- 应用图标、启动图、版本号策略仍需收口

## 2.7 测试与文档同步不足
- 缺少前端单元测试
- 缺少后端接口测试
- 缺少 CI 构建检查
- 根 README / TASKS 需要持续同步实际状态

## 3. 下一阶段优先级

## P0：下一轮必须补齐
这些项决定项目能否从“演示版”进入“可试用版”。

### 3.1 联系人能力补强
- 联系人角色 / 分组 / 优先级
- 区分拨号联系人与短信联系人
- 支持从联系人直接建立通知策略

### 3.2 SOS 历史能力补强
- 时间筛选 / 数量筛选
- 地图化回放
- 通知结果筛选
- 更清晰的事件详情组织

### 3.3 用户 / 设备身份持久化
- 不再硬编码 `u_123`
- 不再硬编码 `android-device-001`
- 首次安装自动生成并持久化用户/设备标识

### 3.4 存储升级
- 本地：从 `localStorage` 升级为更稳的持久化方案（如 SQLite）
- 远端：从内存升级为数据库（SQLite / PostgreSQL）

### 3.5 Android Release 化
- release build
- 签名配置
- 图标 / 启动图 / 包名 / 版本号整理
- 区分 debug 与 release 行为

### 3.6 调试功能隔离
将以下能力放入开发者模式、隐藏入口或 debug build 限定：
- 本地数据面板
- mock 联系人 / 轨迹按钮
- 本地快照导入导出

### 3.7 自动化测试基础设施
- 前端核心逻辑测试
- FastAPI 接口测试
- 基础 CI（至少检查构建是否通过）

## 6. 建议按 3 个阶段推进

## 阶段 A：从“演示版”到“试用版”
目标：让非开发人员也能稳定试用。

建议内容：
1. 用户 / 设备标识持久化
2. 调试功能隐藏
3. Release APK 构建
4. 联系人角色与通知策略补强
5. SOS 历史筛选与地图化增强

### 阶段 A 验收标准
- 不依赖 mock 按钮也能完整操作主要流程
- 普通用户可以管理联系人并完成 SOS 触发
- 可以查看历史 SOS 事件
- 能生成 release 包并稳定安装

## 阶段 B：从“试用版”到“可靠版”
目标：提高数据可靠性与真实场景可用性。

建议内容：
1. 本地持久化升级
2. 远端数据库化
3. 定位刷新与轨迹策略
4. 测试与 CI

### 阶段 B 验收标准
- App 重启 / 后端重启后关键数据不丢失
- 本地与远端模式都有基础自动化验证
- 真实移动测试中能稳定记录轨迹与 SOS 数据

## 阶段 C：从“可靠版”到“产品增强版”
目标：体现差异化价值。

建议内容：
1. 风险区域提示
2. 安全路线推荐
3. 报警事件地图回放
4. 更深层的守护体验功能

---

## 7. 当前最推荐的下一步组合

如果只做一轮迭代，建议优先做这 6 件事：
1. 用户 / 设备标识持久化
2. 本地调试能力隔离
3. Release APK / 签名
4. 联系人角色 / 通知策略补强
5. SOS 历史筛选 / 地图化增强
6. README / TASKS / 测试清单同步更新

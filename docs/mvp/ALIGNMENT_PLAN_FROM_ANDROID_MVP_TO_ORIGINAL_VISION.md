# 从当前 Android MVP 向原始文档能力靠拢：总路线图

更新时间：2026-03-25

## 1. 文档目的

本文用于回答一个明确问题：**当前仓库已经完成的 Android MVP，应该如何在不脱离现实资源与工程约束的前提下，逐步向《神秘组织内部资料》中描述的原始能力愿景靠拢。**

它不是新的需求池，也不是“把原始文档所有功能重新抄一遍”。

本文基于以下事实来源统一规划：
- 原始愿景来源：`神秘组织内部资料.md`
- 当前项目定位与已实现范围：`README.md`
- 现有路线图骨架：`docs/mvp/PROJECT_STATUS_AND_ROADMAP.md`
- 上一轮工程评审结论：`docs/mvp/PROJECT_IMPROVEMENTS_REVIEW.md`
- 差异审查口径：`docs/mvp/DOCUMENTATION_CODE_MISMATCHES.md`
- 当前任务优先级：`docs/mvp/TASKS.md`
- 代码事实：`backend/main.py`

本文的目标有四个：
1. 统一“原始愿景 / 当前 MVP / 下一步工程化补洞”三套叙事；
2. 给出一个**分阶段、可验收、可同步到任务清单**的总路线图；
3. 从大量差距中明确区分：**建议立即做 / 建议延后 / 不建议近期做**；
4. 避免再次出现“文档写得像已经做了，但代码其实还没有”的偏差。

---

## 2. 当前共识：项目不是在“补齐所有愿景”，而是在“先把 MVP 变成可信底座”

### 2.1 原始文档里的愿景范围非常大

《神秘组织内部资料》里给出的首批核心功能不止是报警，还包括：
- 及时报警与追踪定位
- 检测偷拍
- 安全指导
- 安全路线
- AI 辅助（情绪陪伴、伪装声音）

同时它还描述了大量高阶后台能力，例如：
- 前后台精准定位与高频轮询
- 主动报警 + 自动智能报警
- 地理围栏、异常移动、SIM/关机/卸载预警
- 高分贝 / 跌倒 / 碰撞等环境或传感器触发
- 规则引擎、分级响应、多协议上传
- 地图实时渲染、历史回放、位置纠偏
- 远程指令（如静音录音、远程锁机）
- 保活、硬件联动、隐私合规与最小授权可见

这说明原始文档描述的不是一个“轻量演示 App”，而是一个相对完整的个人安全产品愿景。

### 2.2 当前仓库的真实定位已经收缩为 Android MVP 演示闭环

根 README 已经明确，当前项目的真实状态是：
- 已完成一版可运行的 Android MVP；
- 当前主线能力集中在一键 SOS、联系人通知、位置记录与轨迹查询；
- 支持 Android APK 内本地后端模式；
- 已具备基础测试与 CI；
- 下一阶段优先方向不是扩展高级功能，而是先补鉴权/CORS、模块边界、配置统一、API 契约一致性。

这意味着：
- 当前项目**不是原始愿景的完整实现**；
- 当前项目也**不是失败的半成品**；
- 它已经形成了一个可运行、可构建、可测试、可演示的 MVP 底座，但离原始文档中的“完整安全体系”仍有显著距离。

### 2.3 上一轮评审已经给出关键原则：先补工程基线，不先横向堆功能

`docs/mvp/PROJECT_IMPROVEMENTS_REVIEW.md` 已经明确：
- 当前最优先的是工程基线；
- 已完成基础自动化测试与 CI；
- 待继续推进的是远端身份/鉴权/CORS、文件拆分、配置统一、API 契约一致性；
- 一些产品增强项虽然重要，但暂不应排在工程基线之前。

因此，本路线图的核心原则是：

> **先把当前 Android MVP 变成“可信试用底座”，再选择性向原始愿景靠拢；而不是立即把所有高阶能力同时拉进实施范围。**

---

## 3. 规划方法：采用“能力对齐”而不是“功能堆叠”

为了避免路线图失控，本文按三层结构规划：

### 第一层：底座可信度
回答的是：当前 MVP 能不能被真实试用、能不能解释清楚边界、会不会因为基础安全和可靠性问题导致整个产品方向失去可信度。

### 第二层：原始愿景中的高价值补位
回答的是：在大量未实现能力中，哪些最值得回补，且能够明显缩小“原始承诺 vs 当前 MVP”的感知差距。

### 第三层：中长期差异化能力
回答的是：哪些功能确实属于原始愿景，但目前数据、法规、工程成本、误报成本都不适合立刻做。

基于现有文档与代码证据，当前最值得优先对齐原始愿景的 3 个缺口是：
1. **远端身份 / 鉴权 / CORS / 归属校验**
2. **SOS 可靠性补强 + Android 后台限制边界澄清**
3. **地图化回放（SOS 历史 / 轨迹）**

选择这 3 项的原因是：
- 它们都直接影响当前 MVP 是否“可信”；
- 它们都能显著缩小与原始文档“报警、追踪、后台处理、地图回放”的核心认知差距；
- 它们都已在 README / roadmap / TASKS / 评审文档中出现，具备连续证据链；
- 它们比“偷拍检测、AI 伪装声音、规则引擎、远程锁机”等能力更适合作为下一阶段投入。

## 4. 现状基线：当前已经具备什么，仍缺什么

### 4.1 已经具备的 MVP 能力

当前仓库已经具备的主干闭环可以概括为：
- Android 端权限申请与基础引导
- 联系人管理
- SOS 倒计时与取消
- SOS 历史记录与通知详情
- 周期轨迹记录、弱网补发、本地快照导入导出
- 本地后端模式 + 远端 FastAPI + SQLite
- 基础前后端测试与 GitHub Actions CI
- debug / release APK 与 release AAB 构建链路

这意味着当前项目已经达到：
- 可运行
- 可安装
- 可演示
- 可在 Android 上完成本地闭环验证

### 4.2 代码事实确认的核心缺口

结合 `backend/main.py`，当前仍有几个不能回避的事实：

#### 4.2.1 远端安全基线尚未建立
后端当前：
- CORS 仍是 `allow_origins=["*"]`；
- 没有登录、token、session、API key 等正式鉴权机制；
- 数据模型与接口读写仍主要依赖 `userId` / `deviceId` 字段；
- 这意味着远端模式下的归属校验与最小安全边界并未形成。

#### 4.2.2 现有 SOS 通知仍是模拟链路，不是规则化响应系统
`simulate_notify()` 目前仅做：
- 模拟拨号发送
- 模拟短信发送
- 记录发送/跳过日志

它并不等于：
- 分级响应引擎
- 智能告警编排
- 多角色通知策略
- 与原始文档一致的服务器侧规则判断与联动

#### 4.2.3 数据库能力只覆盖 MVP 基线
当前建表只覆盖：
- `emergency_configs`
- `contacts`
- `tracking_points`
- `sos_events`
- `sos_notifications`

这只能支撑 MVP 的基础报警、联系人、轨迹、历史闭环，无法直接支撑：
- 地理围栏
- 异常识别
- 规则引擎
- 远程控制
- 复杂权限体系
- 多终端守护视图

### 4.3 差异审查的关键口径

`docs/mvp/DOCUMENTATION_CODE_MISMATCHES.md` 已明确提醒：
- 文档写了“下一步要做安全收口”，不等于代码已经具备安全能力；
- API 文档能代表当前 MVP API 边界，但不能外推出“更高级能力只是没写文档”；
- 最终路线图必须以**代码已实现 + 文档明确承认未完成**的交集来判断优先级。

## 5. 总体策略：分三类管理能力缺口

### 5.1 建议立即做
这类事项满足以下条件：
- 直接影响 MVP 的可信试用；
- 能明显缩小与原始文档“报警与追踪定位”主线的差距；
- 已有现成文档证据支撑，不需要重新定义方向；
- 投入产出比高于继续横向扩功能。

### 5.2 建议延后
这类事项通常：
- 价值明确，但不阻断当前 MVP 进入下一阶段；
- 依赖前面几个基础能力先落地；
- 若现在就做，容易把资源从工程基线抽走。

### 5.3 不建议近期做
这类事项通常：
- 成本高、误报/合规风险高、数据依赖重；
- 即便做了，也未必比补基础能力更能提升当前版本可信度；
- 与当前 Android MVP 的实际架构距离过远。

---

## 6. 能力分组：立即做 / 延后做 / 近期不做

## 6.1 建议立即做

### A. 远端身份 / 鉴权 / CORS / 归属校验
**为什么立即做**：
- 这是当前从“可演示”走向“可试用”的最小安全基线；
- README、Roadmap、TASKS、评审文档都已把它列为下一阶段高优先事项；
- 代码事实已明确表明当前远端模式仍过于开放。

**与原始愿景的对齐价值**：
- 对齐原始文档中的“仅授权账号可见”“最小权限原则”；
- 为未来的多设备、监护端、远程查看、规则引擎打下归属基础。

### B. SOS 可靠性补强 + Android 后台限制边界澄清
**为什么立即做**：
- 原始愿景的核心不是“页面上有 SOS 按钮”，而是“真实出险时能尽量发得出去、失败时知道怎么补救”；
- 当前 roadmap 和 TASKS 都已把失败恢复、后台限制、前台服务 PoC 决策列为 P0；
- 这决定项目是否会在真实场景下出现“文档承诺大于系统真实能力”的风险。

**与原始愿景的对齐价值**：
- 对齐“持续追踪、异常情况下不中断、联网后补传、后台限制说明”的现实落地版本；
- 让“原始文档中的持续守护愿景”先转化为一个可验证、可解释的 Android 现实边界。

### C. 地图化回放（SOS 历史 / 轨迹）
**为什么立即做**：
- 原始文档强调地图实时轨迹与历史回放；
- 当前 MVP 虽然已记录轨迹与 SOS 历史，但主要还是列表视图；
- 对外演示、事后复盘、功能说服力都明显受限。

**与原始愿景的对齐价值**：
- 这是当前最接近原始“追踪定位”主线、且用户能直观看到价值的补位能力；
- 比继续堆抽象能力更容易形成清晰成果。

## 6.2 建议延后

### D. 联系人角色 / 分组 / 优先级 / 通知策略
价值明确，但建议排在安全基线和失败恢复之后。
原因：
- 它能提升“通知链路编排”的成熟度；
- 但如果远端归属校验、后台限制、SOS 失败恢复都还没收口，先做这类策略性增强会导致主次倒置。

### E. 存储与隐私补强（本地结构化存储、脱敏、快照风险治理）
这是必须做的，但更适合放在“试用版 -> 可靠版”阶段系统收口。
原因：
- 它涉及本地存储升级、敏感字段边界、导入导出提示、可能的迁移逻辑；
- 需要在前面把身份边界和核心数据流先稳定下来。

### F. 风险区域提示
可作为“从求助走向预防”的增强能力，但依赖：
- 地图能力更成熟；
- 风险源数据可获取；
- 误报容忍度与提示频率可控。

### G. 安全路线推荐（占位版）
原始文档中很重要，但当前不适合前置。
原因：
- 依赖稳定地图能力、风险数据、推荐依据可解释性；
- 一旦做得过早，容易沦为看起来高级、实际可信度不足的展示项。

## 6.3 不建议近期做

以下能力保留为中长期方向，但不建议近期进入主线实施：

### H. 偷拍检测
原因：
- 技术路径、误报漏报、设备差异、隐私合规都非常复杂；
- 与当前 MVP 的报警与轨迹闭环技术栈不连续。

### I. AI 情绪陪伴 / AI 伪装声音
原因：
- 差异化强，但不属于当前 MVP 可信度短板；
- 容易分散团队在安全基线与可靠性问题上的投入。

### J. 规则引擎 / 自动智能报警全家桶
包括地理围栏、异常移动、强制确认、环境与传感器自动触发等。
原因：
- 需要稳定后台能力、误报治理、配置体系、告警策略、长期测试数据；
- 若在当前阶段硬做，很可能出现“演示像高级能力、真实不可控”的情况。

### K. 远程录音 / 远程锁机 / 强控制指令
原因：
- 合规风险高；
- 需要严格鉴权、审计、权限模型、设备兼容性验证；
- 与当前 MVP 主线距离过大。

### L. SIM/关机/卸载预警、硬件联动、长时保活
原因：
- 依赖更强的系统级/硬件级能力；
- Android 厂商差异大，可靠性难统一承诺；
- 应在完成 Android 限制边界研究后再决定是否有任何 PoC 价值。

## 7. 分阶段总路线图

本文建议采用四个阶段，而不是把所有内容堆进单一 roadmap 条目。

## 阶段 0：统一事实与约束（当前立即开始）

### 阶段目标
在正式进入功能实施前，先把“当前是什么、下一步做什么、不做什么”统一成同一口径，避免再次出现文档与代码叙事脱节。

### 范围
- 统一 roadmap、TASKS、README 与本路线图的优先级叙事
- 固化 3 个最高价值缺口：
  1. 鉴权/CORS/归属校验
  2. SOS 可靠性 + Android 后台边界
  3. 地图化回放
- 固化“不建议近期做”的能力范围

### 依赖
- 已有 docs/mvp 差异审查与评审结论
- 当前 README、TASKS、Roadmap 作为既有依据

### 风险
- 若文档不同步，后续执行会重新回到“有人按工程基线做，有人按原始愿景堆功能”的分裂状态。

### 验收标准
- 团队后续执行时，不再把安全方向误认为已实现能力；
- P0/P1/P2 与“立即做 / 延后 / 近期不做”口径一致；
- 文档能直接作为后续任务拆分依据。

### 排优先级原因
这是所有实施前的对齐动作，成本最低、收益最高。

---

## 阶段 1：从 Android 演示版到可信试用底座

### 阶段目标
让当前 MVP 从“可展示 Demo”升级为“边界明确、风险可控、可以组织小范围试用”的版本。

### 重点范围

#### 1) 远端身份 / 鉴权 / CORS / 归属校验
建议落地内容：
- 明确 MVP 级身份方案（无需一步到位做完整账号体系）
- 定义请求头、设备标识、用户归属与跨用户拦截规则
- CORS 从全开放切换到按环境白名单配置
- 文档、接口约定、错误语义同步更新

#### 2) SOS 可靠性补强
建议落地内容：
- 明确定位失败、超时、短信失败、拨号失败、联网失败的用户可见反馈
- 形成 SOS 状态机与补救提示
- 对断网补发、恢复联网后的行为给出清晰验收口径

#### 3) Android 后台限制与前台服务边界
建议落地内容：
- 输出 Android 10+/13+/14+ 的前台服务、后台定位、Doze、被杀进程等约束说明
- 决定“是否做前台服务 PoC，以及仅限哪类短时场景”
- 明确哪些能力只验证、哪些能力本轮不承诺

#### 4) 地图化回放（最小版）
建议落地内容：
- 从单次 SOS 历史进入地图回放
- 支持轨迹点位、时间轴、关键节点查看
- 先服务复盘、说明与演示，不先做复杂智能分析

### 依赖
- 当前已有轨迹写入、SOS 历史、后端 SQLite 存储
- 当前已有基础测试与 CI
- 当前已有文档中明确的 P0 骨架

### 风险
- 若身份方案过重，会拖慢整体进度；
- 若后台守护表述过于乐观，会再次制造承诺风险；
- 若地图能力选型过早做复杂化，容易偏离“最小回放”目标。

### 验收标准
- 非白名单来源受限，跨归属读写被拒绝；
- SOS 在失败路径中不再静默；
- 至少完成一轮 Android 10+/13+ 真机边界验证记录；
- 事件详情可进入地图查看轨迹与关键点；
- README / TASKS / Roadmap / API 文档对远端访问约束与后台能力边界口径一致。

### 排优先级原因
这是当前版本最能缩小“原始愿景核心主线”与“当前 MVP 实际可用性”差距的一阶段。

---

## 阶段 2：从可信试用底座到可靠版

### 阶段目标
解决数据可靠性、隐私边界与长期维护成本问题，让系统更接近真实场景使用基础。

### 重点范围

#### 1) 存储与隐私补强
- 本地持久化从简单 Key-Value 继续升级到结构化存储
- 明确哪些数据必须持久化、哪些需要脱敏、哪些只能开发态暴露
- 对快照导入导出、清数据、敏感字段显示增加风险提示与约束

#### 2) 测试与 API 契约深化
- 增加失败场景、边界场景、鉴权/CORS 回归测试
- 补本地后端与远端后端的一致性契约校验
- 把阶段 1 新增的边界都纳入自动化回归

#### 3) 模块拆分与配置统一
- 继续拆分 `backend/main.py`、`frontend/src/App.jsx` 等大文件
- 收敛版本号与环境变量入口
- 降低后续继续演进时的耦合成本

### 依赖
- 阶段 1 已形成稳定的身份边界与状态语义
- 阶段 1 已明确 Android 能力边界，不再反复变更

### 风险
- 若在阶段 1 之前做大规模存储迁移，容易和鉴权、契约、错误语义重叠返工；
- 若没有测试兜底，阶段 2 重构会带来回归风险。

### 验收标准
- App 重启 / 后端重启后关键数据与关键状态不异常丢失；
- 本地 / 远端模式关键接口契约一致性可被自动校验；
- 关键敏感数据边界、快照风险提示、清理策略形成文档与实现闭环；
- 前后端主要大文件停止继续膨胀。

### 排优先级原因
阶段 2 解决的是“可试用之后能不能稳定迭代”的问题，优先级应低于阶段 1 的可信度补洞，但高于任何新的大功能扩张。

## 阶段 3：向原始愿景做有限增强

### 阶段目标
在底座可靠后，选择少量与原始文档高度相关、且能形成差异化感知的增强能力。

### 建议范围
- 联系人角色 / 分组 / 优先级 / 通知策略
- 风险区域提示
- 安全路线推荐（先占位版，再判断是否可持续）
- 地图回放增强（筛选、标签、事件聚合）

### 依赖
- 需要阶段 1 的地图基础与身份归属能力
- 需要阶段 2 的存储、测试、配置与模块边界稳定

### 风险
- 风险区域与路线推荐若缺少数据依据，会削弱可信度；
- 联系人策略若过度复杂，会增加配置负担。

### 验收标准
- 新增能力能够解释“为什么提示 / 为什么推荐 / 为什么先通知某联系人”；
- 不引入明显高于当前团队承载能力的数据治理负担；
- 文档不夸大仍处于实验状态的能力。

### 排优先级原因
这类能力能让产品更像原始愿景中的“主动守护工具”，但前提是底座已经稳住。

### 阶段目标

## 8. 为什么是这三个优先缺口，而不是其他能力

## 8.1 为什么优先“鉴权/CORS/归属校验”
因为它是当前最明确、最可证伪、最不能继续拖延的缺口：
- 代码里已明确存在全开放 CORS；
- 远端接口缺少真正鉴权；
- 这会直接影响联系人、轨迹、SOS 历史等敏感数据边界；
- 不先补这个，任何“监护端、远程查看、策略通知”都没有可信基础。

## 8.2 为什么优先“SOS 可靠性 + Android 后台边界”
因为安全产品最怕的不是“功能少”，而是“关键时刻用户误以为能工作，结果边界不清”。
- 当前已经有位置采样、弱网补偿、精度提示，但失败恢复路径与后台边界仍需收口；
- 这直接关系到真实使用信任；
- 也是将原始文档里的“持续追踪”能力转换为现实 Android 能力边界的必要步骤。

## 8.3 为什么优先“地图化回放”
因为它是原始“报警与追踪定位”主线中最直观、最能被用户感知、最适合在当前技术底座上继续前进的一步。
- 当前数据已经有；
- 只是展示与复盘能力不足；
- 这是少数既能增强产品感知，又不必立刻引入高复杂智能判断的能力。

## 10. 最终结论

当前仓库最合理的判断不是“离原始愿景太远所以需要全面重做”，而是：

> **它已经完成了一个 Android 安全 MVP 的最小闭环，但接下来必须先补可信度，再做能力靠拢。**

因此，面向《神秘组织内部资料》的总路线图建议明确为：

### 建议立即做
1. 远端身份 / 鉴权 / CORS / 归属校验
2. SOS 可靠性补强 + Android 后台限制边界澄清
3. 地图化回放（SOS 历史 / 轨迹）

### 建议延后
1. 联系人角色 / 分组 / 优先级 / 通知策略
2. 存储与隐私补强
3. 风险区域提示
4. 安全路线推荐（占位版）

### 不建议近期做
1. 偷拍检测
2. AI 情绪陪伴 / AI 伪装声音
3. 规则引擎与自动智能报警全家桶
4. 远程录音 / 锁机 / 强控制能力
5. SIM/关机/卸载预警、硬件联动、长时保活

如果后续只允许集中投入一轮主线迭代，那么最推荐的组合仍是：
- **先补安全基线**
- **再补 SOS 失败恢复与 Android 能力边界**
- **随后补地图化回放作为最有感知的能力对齐项**

这条路径最符合当前代码现状、现有 docs/mvp 结论，以及“从 Android MVP 逐步向原始能力愿景靠拢”的现实节奏。
